🌌 PangeaOS とは (Vision)

PangeaOSは、OSの歴史的アーキテクチャ（Multics, UNIX, Plan 9, L4マイクロカーネル）を深くリスペクトしつつ、現代のサイバーセキュリティにおける「攻撃者の視点」を先回りした防御的設計（Offensive Defense）を組み込んだ次世代OSです。既存のOSが抱える「マザーボードやファームウェア（UEFI/ACPI）の挙動に対する盲信」を完全に破壊し、純粋なソフトウェアによるハードウェアの絶対的支配を確立することを目指します。

v0.0.4では、OSの歴史から「ファイルシステム（VFS）」という概念そのものを破棄しました。新設された Single-Level Store (SLS) アーキテクチャにより、永続データはすべて巨大な「直交的永続メモリ空間」にオブジェクトとして直接マッピングされます。SIP（プロセス）は `open()` や `read()` といった非効率なシステムコールを一切使わず、割り当てられた仮想メモリアドレスを直接読み書きするだけで、背後のNVMe/SSDへのフラッシュがOSによって透過的かつ自動的に行われます。これにより、PangeaOSはデータ管理においても究極のゼロコピー・特異点を達成しました。




🎯 究極のハッキング・アーキテクチャ (Core Architecture)

一般的なOS開発のセオリーを意図的に逸脱した、以下の独自アーキテクチャを採用しています。

1. Ring 0 JIT Compiler & Zero-Cost MBC 【v0.0.1-9-1 目玉機能 ✨】

PangeaOSの核となるのは、特権空間（Ring 0）の内部で稼働する超高速な動的ネイティブコード生成エンジンです。このJITコンパイラは、外部の未検証コード（ASHバイトコード）を、実行時にx86_64の純粋なマシン語へ直接トランスパイルします。v0.0.1-9-1では、単一パケットの検査・チューリング完全なループ・状態保持・Ring 0 FFI に加え、**Constant Blinding (JIT Spraying 防御)** と **FFI へのグローバル Gas リミット** を備え、悪意のある入力からカーネルを完全に保護します。

最大の特異点は「境界チェックの完全ブランチレス化 (MBC)」です。領域外メモリアクセスを防ぐため、実行時の分岐（if 文）を挿入するのではなく、トランスパイル段階で物理的なビットマスク（and r8, 0x3F）をスクラッチレジスタを介して強制挿入します。これにより、コンパイル時にオフセットが不明な動的パケットアクセスであっても、OSをクラッシュさせることなく、C言語のハードコードと同等以上の光速実行を維持します。さらに、フォワードジャンプは常に命令境界へ正確に着地するようバイトオフセットを計算するため、Control Flow Integrity (CFI) がハードウェアレベルで保証されています。

JITコンパイラは、専用の4KBページをアロケータから確保し、コンパイルされたマシンコードをそこに配置します。このページはW^X Enforcerによって保護され、実行中に書き換えられることを防ぎます。また、CallExt命令を通じて、_rdtsc()による高精度タイマー取得や、serial_println!によるデバッグ出力といったカーネルヘルパー関数を直接呼び出すことが可能です。

2. W^X (Write XOR Execute) Memory Enforcer 【v0.0.1-7 目玉機能 ✨】

攻撃者の視点（エクスプロイト手法）を根絶するための、究極のページテーブル動的制御機構です。JITがマシン語を生成するバッファ（ヒープ領域）は、生成中は「書き込み可能・実行不可（RW + NX）」として保護されます。コンパイル完了直後、VMMに介入してTLBをフラッシュし、特権レベルで「読み取り専用・実行可能（RX）」へとページ属性をフリップ（Seal）させます。これにより、JITコードの実行中に悪意のあるコードが注入されることを物理的に防ぎます。

また、メモリ解放（Drop）の直前に再び「RW + NX」へと浄化（リストア）することで、再利用時のカーネルパニックやバッファオーバーフローを突いたシェルコード実行をアーキテクチャ・レベルで完全に撲滅しています。このメカニズムは、AshJit::seal()およびDrop for AshJitの実装によって実現されており、x86_64::structures::paging::Mapper::update_flagsを用いてページテーブルエントリのフラグを動的に変更し、TLBをフラッシュすることでCPUに新たなセキュリティ境界を強制認識させます。

3. Software-Isolated Processes (SIPs)

Microsoft Researchの「Singularity」OSの概念を現代のRustで昇華させました。SIPsは、ハードウェアの特権レベル（Ring 3）や仮想メモリの分離機能（MMU）に一切頼らず、Rustの「所有権システム」と「型システム」のみを保護境界として利用します。これにより、カーネル空間（Ring 0）に複数の独立プロセスを同居させながら、他プロセスによるメモリ破壊を数学的に保証・防止します。ページテーブル切り替えのオーバーヘッドは完全にゼロです。

各SIPは一意のSipIdを持ち、SipEnvというケイパビリティ・トークンを通じて外部環境とやり取りします。SipEnvはCloneを実装しないことで、所有権の移動のみを許容し、データレースをコンパイル時に物理的に防ぎます。SIPは非同期タスク（Future）として実装され、カーネルのエグゼキュータによってゼロコストでネイティブ実行されます。これにより、コンテキストスイッチの概念を破棄し、極めて効率的な並行処理を実現します。

4. Ownership-Driven Zero-Copy IPC

共有メモリによるデータレースの火種を根絶した「契約ベースのチャネル」です。プロセス間通信においてデータのコピー（memcpy）は一切発生しません。送信プロセスがペイロードをチャネルに send した瞬間、コンパイラがポインタの「所有権」を受信側に物理的に移譲（Move）し、以後の送信側からのアクセスを遮断します。これにより、データの一貫性と安全性を保証しつつ、最高のパフォーマンスを実現します。kernel/src/sip/ipc.rsで実装されるチャネルは、SenderとReceiverのペアを通じて、安全かつ効率的なメッセージパッシングを提供します。

5. Hybrid Async-Interrupt Engine

レガシーな 8259 PIC を物理的破棄（0xFF マスク）し、Local APIC へ移行しました。Rustの Future および Waker アーキテクチャを Ring 0 空間に持ち込み、ハードウェア割り込みをトリガーとして動作する 「完全協調型 Async-Await タスクスケジューラ」 を完成させました。これにより、割り込み処理と非同期タスクの実行がシームレスに統合され、リアルタイム性と応答性の高いシステムを実現します。kernel/src/scheduler.rsには、スレッド管理とタスクスケジューリングのロジックが実装されており、main.rsではSimpleExecutorとTaskを用いてSIPが非同期に実行される様子が示されています。

6. True Mesh Allocator (Size-Class メモリ再利用)

外部依存ゼロでフルスクラッチ開発した Size-Class ベースの Segregated Free List アロケータです。kernel/src/allocator/mesh.rsに実装されており、8〜2048バイトのオブジェクトサイズに応じてメモリを動的に切り出し、解放時に即座にフリーリストへ還元します。これにより、メモリフラグメンテーションを最小限に抑え、効率的なメモリ再利用を実現します。Ring 0 空間で Box, Vec, String をメモリリークの恐怖なしに無制限に利用可能です。Mesh Allocatorは、BLOCK_SIZES配列で定義されたサイズクラスに基づいてメモリブロックを管理し、必要に応じてBumpAllocatorをフォールバックとして使用します。

7. Ring 0 Exclusive GDT & True IDT Barrier (絶対防壁)

ユーザー空間への依存を完全に捨て去り、Ring 0専占のGDT（Global Descriptor Table）とTSS（Task State Segment）を構築しました。IST（Interrupt Stack Table）を有効化し、致命的なダブルフォルト発生時も無傷の専用スタックへ逃げ込みシステムクラッシュを物理的に阻止します。これにより、OSの安定性と堅牢性を極限まで高めています。kernel/src/gdt.rsとkernel/src/interrupts.rsにこれらのメカニズムが実装されています。

9. Zero-Stack Double Buffering

描画エンジンはスタックメモリを一切消費せず、すべてカーネルの静的メモリ領域（.bss）に直接マッピングされています。超高速クリアと物理VRAMへの一撃転送によりフリッカーを完全に消滅させました。これにより、グラフィックス処理におけるパフォーマンスを最大化し、スムーズな描画を実現します。

10. True SMP Ignition & Multi-Core Preemption 【v0.0.3-2 目玉機能 ✨】

Limine の MpRequest を利用してすべての Application Processors (AP) をセーフティに初期化し、各コアは独自の GDT (Global Descriptor Table) と IDT (Interrupt Descriptor Table) をロードして保護モードに入ります。これにより、トリプルフォールトの危険を完全に排除しました。各 AP は独立したエグゼキュータを起動し、SIP (Software-Isolated Processes) を並行して物理実行します。プロセス間の通信はゼロコピーの IPC チャネルによって行われ、完全なマルチコアのポテンシャルを引き出します。また、未使用のインポートやコードの警告をゼロにし、極めてクリーンなコードベースを維持しています。




🛠 開発環境 (Setup & Build)

本プロジェクトは、ホストOSのグローバル環境を一切汚染しないよう Nix Flakes によって完全な Determinism（再現性）を保証しています。開発者間のコンパイラバージョンの差異によるトラブルは発生しません。

Prerequisites (前提条件)

ホストマシンに Nix パッケージマネージャ がインストールされており、Flakes 機能が有効になっていること。

ビルドと実行 (Build & Deployment)

Nixの開発シェルに入った状態で make を叩くだけで、環境変数の浄化、Rustカーネルのコンパイル、Limine MBRブートローダの注入、ISOイメージの生成、QEMUエミュレータの起動までが一撃で完遂します。

Bash


# 1. 開発環境に入る
nix develop

# 2. ビルド＆QEMU起動
make

### 🚀 Phase 5: ハードウェア支援による「物理的隔離」とレガシー互換 (The 0.0.2 Series)

**v0.0.2 系（The 0.0.2 Series）の設計思想：**
既存のオペレーティングシステム（Linux等）は、セキュリティと安定性を「Ring 3（ユーザーモード）への降格」と「巨大なページテーブル（CR3）による仮想メモリ空間の分離」に依存しています。しかし、このアプローチはコンテキストスイッチ（CR3の書き換え、TLBフラッシュ、権限レベルの遷移）という莫大なパフォーマンスの犠牲を伴います。
PangeaOS の v0.0.2 系は、この常識を打ち破る **「Ring 0 SFI (Software Fault Isolation)」** の完成形を目指します。CPUの特権モード（Ring 0）にとどまったまま、SMEP/SMAPやMPK（物理保護）、CHERI（ファットポインタによる領域保護）、そして µFork（CR3を介さないプロセス複製）といった最新のハードウェア・ソフトウェアハイブリッド技術を組み合わせることで、「ユーザーランドよりも安全で、ベアメタルと同等に速い」特異点アーキテクチャを構築します。

ソフトウェア上の「特異点 (Singularity)」である v0.0.1 系に続き、このフェーズでは最新のCPUハードウェア支援技術をRing 0で直接操作し、真のコンパートメント化（細分化された隔離）を実現します。

- [x] **v0.0.2-1: SMEP / SMAP / PKU 機能の初期化**
  - **SMEP / SMAP 有効化**: カーネル（Ring 0）から意図せずユーザーレベル（または無効なデータ）ページを実行・アクセスするのを物理的に防ぐハードウェア保護の有効化。
- [x] **v0.0.2-2: CHERI (制限境界付きポインタ) アーキテクチャの概念統合**
  - 現在ソフトウェアベースで行っている JIT MBC (ブランチレス境界チェック) をハードウェアにオフロード。ポインタ自体に「アクセス権限」と「境界長」を持たせ、Rustの `unsafe` ブロック内であっても領域外アクセスをハードウェアレベルで完全に阻止するコンパートメント化の実現。
- [x] **v0.0.2-3: µFork (マイクロフォーク) プロセスメカニズムの確立**
  - 巨大なUnixの `fork()` (ページテーブル全体の走査とCoW) を捨て去り、必要なメモリ領域のみをディープコピーしつつJIT実行コードをArcで共有する、極めて軽量かつセキュアなプロセス生成基盤（AshProcess）の構築。隔離されたサンドボックスをTLBミスなしに瞬時にクローン可能に。
- [x] **v0.0.2-4: Ring 0 POSIX エミュレーションとレガシーバイナリ互換**
  - 通常 Ring 3 から発行されるシステムコール (`syscall`) を Ring 0 空間内でトラップし、特権リング遷移（sysret）を行わずに超低レイテンシでエミュレート。既存の Linux バイナリなどを PangeaOS の SIPs 内部で安全に稼働させる「サンドボックス型互換レイヤー」の実装。
- [x] **v0.0.2-5: PKS (Supervisor Protection Keys) ドメインの完全なアイソレーション**
  - x86_64 クレートの制限を迂回し、手動の CR3 ページテーブル走査により PTE (59〜62bit) へ Protection Key を直接スタンプする機構を実装。
  - SIPの実行直前に `MSR_PKRS` を操作し、自身のキーとカーネル以外の全メモリへのアクセスをハードウェアレベルで遮断する「完全なるゼロコスト・ドメイン隔離」の達成。

### 🚀 Phase 6: 真のマルチコア並列実行と高並行スケジューリング (The 0.0.3 Series)

**v0.0.3 系（The 0.0.3 Series）の設計思想：**
Ring 0 環境下での高度なアイソレーション（SFI, PKS）を達成した PangeaOS は、遂に複数の物理プロセッサ・コアを極限まで活用する「真の対称型マルチプロセッシング（SMP）」の段階へと移行します。これまではBSP（Boot Strap Processor）による時分割プリエンプション（シングルコア非同期スケジューリング）で動作していましたが、本フェーズではすべてのコア（AP）を完全に覚醒させ、独立したSIP（Software-Isolated Processes）を複数の物理コア上で同時にネイティブ実行します。さらに、コア間で発生するデータの共有や同期のコストをゼロにするため、「所有権の移動」に基づくゼロコピーIPC通信をハードウェアレベルで完全に並列稼働させます。

- [x] **v0.0.3-1: 真のSMPマルチコア実行の確立とAP覚醒**
  - Limine の MpRequest を利用して駐車（Halt）状態のAPにウェイクアップ信号を送信。各コアが独自の GDT と IDT をロードして保護モードに入り、トリプルフォールトを防ぎつつマルチコアをセーフティに初期化。
  - シングルコア環境の時分割スケジューラを廃止し、独立したタスクを物理的に異なるコア（AP 1, AP 2）上で同時に走らせる「完全並列実行アーキテクチャ」の完成。
- [x] **v0.0.3-2: マルチコア間における所有権駆動のゼロコピーIPC実証**
  - AP 1 上の SIP Alpha と AP 2 上の SIP Beta の間で、Rust の Channel 機構をベースとしたロックフリーのメッセージパッシングを実証。複数コアが互いにメモリを破壊することなく、完全に非同期かつ無駄なデータコピーなしに情報を共有する基盤の完成。
- [x] **v0.0.3-3: Cross-Core TLB Shootdown によるマルチコアW^X絶対防壁**
  - JITコンパイル直後のページ属性フリップ（RW → RX等）において、変更を行ったコアが他の全アクティブコアへIPI（プロセッサ間割り込み）を発行。
  - すべてのコアが古いTLBエントリを破棄し、ACKを返すまでスピンロックで厳密に同期するシュートダウン・メカニズムを確立し、マルチコア特有のキャッシュの隙を突くTOCTOU攻撃を完全に遮断。

### 🛡️ Phase 7: リング0 特異点アーキテクチャの絶対防壁化 (Advanced Security & Resilience)
マルチコア化とRing 0 SFIを達成したPangeaOSを、あらゆる攻撃ベクトル（DoS、サイドチャネル、リソース枯渇）から完全に保護するための次世代セキュリティ機能の実装フェーズ。

- [x] **v0.0.3-4: IPC バックプレッシャー機構と OOM DoS 防御**
  - 無限に成長しうるゼロコピーIPCチャネルを Bounded Channel（容量制限付き）へ移行し、エグゼキュータレベルでの非同期背圧（Backpressure）を実装。悪意のあるタスクによるRing 0メモリの食いつぶしを物理的に防ぐ。
- [x] **v0.0.3-5: PKS Key Multiplexing (動的キー再割り当て機構)**
  - ハードウェア仕様上「15個」しか存在しないPKS（Supervisor Protection Keys）の制限を突破するため、スケジューラのコンテキストスイッチと連動してアクティブコアにのみ動的にキーを再割り当てするリサイクル機構の実装。
- [x] **v0.0.3-6: Ring 0 Shadow Stack (Intel CET) による ROP 攻撃の撲滅**
  - ソフトウェア的な制御フロー整合性（CFI）に加え、ハードウェア支援による裏スタック（Shadow Stack）を有効化し、カーネル空間での Return-Oriented Programming を物理的に不可能にする。
- [x] **v0.0.3-7: マルチコア・サイドチャネル攻撃（Spectre/Meltdown等）の緩和**
  - JITコンパイラにおける Speculative Load Hardening（`lfence` の自動挿入）および Intel CAT (Cache Allocation Technology) を用いた L3 キャッシュのSIP単位での物理分割。
- [x] **v0.0.3-8: 究極のアーキテクチャ (Intel IBT & eBPF-style Static Verifier)**
  - 全ての間接分岐先に `ENDBR64` を強制し JOP/COP 攻撃を無力化する **Intel IBT** の実装、および、JITコンパイル前に命令グラフを静的解析し、実行時オーバーヘッドなしに停止性と安全性を数学的に証明する **Static Verifier** の導入。（さらに、即値をメモリ上で難読化する Constant Blinding は先行実装済み）

### 🌌 Phase 8: VFSの完全廃止と「データ管理の特異点」
- [x] **v0.0.4: 単一レベルストア (SLS) の実現とファイルシステム概念の破棄**
  - `open()` / `read()` / `write()` などのシステムコールや階層型ディレクトリ構造を完全に廃止し、永続データはすべて「巨大な仮想メモリ空間に直接マップされたオブジェクト」として扱う（直交的永続性）アーキテクチャの基盤を実装。
- [x] **v0.0.4-1: Lock-Free Object Directory & Demand Paging**
  - オブジェクトID (OID) から仮想アドレスへのマッピングを管理するロックフリー構造を実装。アクセス時の `#PF` (Page Fault) をトリガーとした透過的なストレージからの遅延読み込み（Demand Paging）を実装し、完全な Single-Level Store を確立。
- [x] **v0.0.4-2: SLS セキュリティ強化 (Object Bound Checking)**
  - Demand Paging による「ページフォルト爆弾（物理メモリ枯渇 OOM DoS 攻撃）」を無力化するため、ページフォルト発生時に「生成済みオブジェクトの正規の範囲内か」をロックフリーディレクトリのメタデータ（サイズ）と照合して厳密に検証するバウンドチェック機構を実装。範囲外の不正アクセスは即座にパニック（SIPキル）として処理。
- [x] **v0.0.4-3: ケイパビリティベースのアクセス制御 (Object Capabilities)**
  - JIT環境（ASH Engine）からSLSオブジェクトをマッピングする際、単なるOIDの指定だけでなく、カーネル固有のシークレットキーとOIDを暗号論的にハッシュ化した `Capability Token` の提示を強制。不正なトークンによるマッピング要求を遮断し、完全な CHERI-like なメモリアクセス制御を実現。
- [x] **v0.0.4-4: Lock-Free Directory DoS 防御 (Anti-Exhaustion)**
  - 悪意のあるSIP（プロセス）がランダムなOIDでオブジェクトを大量生成し、システム全体のロックフリーディレクトリ（ハッシュテーブル）を枯渇させる Algorithmic Complexity Attack を防ぐため、SIPごとに作成可能なオブジェクト数に厳しいクオータ（上限）を設けるバックプレッシャー機構を実装。上限に達したSIPの割り当て要求は拒絶される。
- [x] **v0.0.4-5: 透過的メモリ暗号化 (Transparent Memory Encryption: TME)**
  - SLSデータが物理ストレージに書き出される際、および Demand Paging によってメモリに読み込まれる際に、透過的にメモリ上のデータを暗号化・復号化する TME シミュレーション・エンジンを実装。物理的なコールドブート攻撃や NVMe/ストレージの盗難から完全なデータ保護（FDE/TME）を確立。

### 🌐 Phase 9: 超高速ネットワークスタックと Ring 0 Web サーバ
- [x] **v0.0.5: PCI バス列挙と Intel E1000 ギガビットNICドライバ基盤**
  - PCI Configuration Space のスキャンと、Intel E1000 Gigabit Ethernet カードの検出・MMIO（Memory Mapped I/O）マッピング。
- [x] **v0.0.5-1: Zero-copy DMA 受信・送信リングの構築**
  - NICとカーネルメモリを直結。カーネル/ユーザー間のメモリコピーを排除した DMA リングバッファの割り当て。
- [x] **v0.0.5-2: 非同期ネットワーク割り込みとスケジューラ統合**
  - ハードウェア割り込み（IRQ）を IDT に登録し、NICからのパケット到着イベントを非同期に検知。E1000の割り込みマスク（IMS）とクリア（ICR）を制御し、パケット処理をエグゼキュータにシームレスに引き継ぐ基盤を確立。
- [x] **v0.0.5-3: Ring 0 TCP/IP プロトコルスタック（Ethernet / ARP 解析）**
  - 割り込みハンドラから直接パケットのポーリングを行い、生のバイト列からゼロコピーで Ethernet ヘッダおよび ARP (Address Resolution Protocol) ヘッダを解析する処理を実装。
- [x] **v0.0.5-4: ゼロコピー送信基盤および ARP / ICMP (Ping) 応答機構**
  - NICへのゼロコピー送信リング (TX) バッファを割り当て実装。ARP Requestに対する MAC アドレスの自動応答、および ICMP Echo Request に対する Ping 応答 (Pong) をカーネル単独で実現。
- [x] **v0.0.5-5: Ring 0 TCP スタックと Singularity Web Server**
  - TCPプロトコルの3ウェイ・ハンドシェイク（SYN -> SYN-ACK）をインターセプトして処理する軽量ステートレスTCPスタックを実装。
  - HTTPリクエスト（GET）に対して、カーネルの物理メモリ領域から直接 HTML データを生成・構築し、NIC に DMA 送信する「コンテキストスイッチ・ゼロ、コピー・ゼロ」の究極のWebサーバを稼働。
